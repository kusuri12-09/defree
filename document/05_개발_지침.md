# defree — 개발 지침 및 코딩 컨벤션

---

## 1. 프로젝트 구조 및 환경

### 1-1. 모노레포 아키텍처

본 프로젝트는 **Turborepo**와 **pnpm workspaces**를 활용한 모노레포 구조로 관리됩니다.

```
defree/
├── apps/
│   ├── api/                     # NestJS 백엔드 (REST API + 스케줄러)
│   └── web/                     # React PWA (프론트엔드)
├── packages/
│   ├── shared-types/            # 공통 DTO / 인터페이스 (백엔드, 프론트엔드 공유)
│   ├── ocr/                     # OCR + LLM 정규화 파이프라인
│   └── notification/            # Discord / Slack / FCM 웹훅 모듈
├── docker-compose.yml           # PostgreSQL + Redis 등 로컬 개발 인프라
├── turbo.json                   # Turborepo 파이프라인
└── pnpm-workspace.yaml          # pnpm 워크스페이스 정의
```

### 1-2. 패키지 매니저

- 반드시 **pnpm**을 사용합니다. (npm, yarn 사용 금지)
- 종속성 설치: `pnpm install`
- 루트 레벨 스크립트 실행: `pnpm build`, `pnpm dev` 등

## 2. 공통 코딩 컨벤션 (TypeScript)

### 2-1. 네이밍 컨벤션

- **파일/폴더명**: `kebab-case` (예: `user-profile.component.tsx`, `auth.controller.ts`)
- **변수/함수명**: `camelCase` (예: `getUserInfo`, `isLoggedIn`)
- **클래스/인터페이스/타입**: `PascalCase` (예: `UserDto`, `AuthService`)
- **상수**: `UPPER_SNAKE_CASE` (예: `MAX_RETRY_COUNT`, `DEFAULT_PAGE_SIZE`)

### 2-2. 타입스크립트

- `any` 타입의 사용은 엄격히 금지합니다. 부득이한 경우 `unknown`을 사용하고 타입 단언을 진행합니다.
- 프론트엔드와 백엔드가 공유하는 타입은 반드시 `packages/shared-types`에 정의하고 임포트하여 사용합니다.

### 2-3. 포매팅 및 린팅

- **Prettier**: 코드 포매팅의 단일 출처로 사용합니다. (싱글 쿼트, 후행 콤마 필수 등)
- **ESLint**: 정적 분석을 통해 코드 품질을 유지합니다. 커밋 전 린트 에러를 반드시 해결해야 합니다.

## 3. 백엔드 개발 지침 (apps/api)

### 3-1. 기술 스택

- **프레임워크**: NestJS
- **데이터베이스**: PostgreSQL 16 (로컬 개발 시 Docker Compose 사용)
- **ORM**: TypeORM 또는 Prisma (팀 협의에 따름, 명세상 PostgreSQL 관계형 활용)
- **캐시**: Redis

### 3-2. 아키텍처 패턴

- NestJS의 기본 구조인 `Controller` - `Service` - `Repository` 패턴을 준수합니다.
- 비즈니스 로직은 철저히 `Service` 계층에 캡슐화합니다.
- `Controller`는 요청/응답 형식과 유효성 검증(DTO, ValidationPipe)만 담당합니다.

### 3-3. 데이터베이스 및 스키마

- 테이블명 및 컬럼명은 데이터베이스 명세에 따라 `snake_case`로 통일합니다.
- 모든 시간 데이터(`createdAt`, `updatedAt`)는 UTC 기준으로 저장되며, 응답 시 클라이언트(KST)에서 변환하도록 설계합니다. (타입: `TIMESTAMPTZ`)

### 3-4. 외부 API 연동 및 알림

- **OCR 및 GPT**: `packages/ocr` 모듈을 통해 분리하여 호출합니다. 응답 속도를 고려해 배치 처리 또는 비동기 처리를 우선합니다.
- **알림 스케줄러**: `@nestjs/schedule`을 사용하며, 알림 발송 로직은 `packages/notification`을 활용하여 모듈화합니다.

## 4. 프론트엔드 개발 지침 (apps/web)

### 4-1. 기술 스택

- **프레임워크**: React (PWA 지원)
- **상태 관리**: 서버 상태는 React Query(또는 SWR), 클라이언트 전역 상태는 Zustand (또는 Jotai) 사용
- **스타일링**: TailwindCSS (선호) 또는 Styled-Components

### 4-2. 컴포넌트 설계

- **기본 단위**: 아토믹(Atomic) 디자인 패턴 또는 도메인 기반 디렉터리 구조를 활용해 재사용성을 높입니다.
- **비즈니스 로직 분리**: UI 컴포넌트 내에 복잡한 비즈니스 로직 작성을 지양하고, Custom Hook으로 분리합니다.

### 4-3. PWA 및 오프라인/알림 지원

- 모바일 브라우저 중심의 사용자를 위해 **Service Worker**를 철저히 구성합니다.
- FCM을 통한 Web Push 알림 기능을 구현하며, 권한 요청은 사용자가 앱의 가치를 충분히 인지한 시점(예: 첫 영수증 스캔 직후)에 진행합니다.

## 5. 깃(Git) 브랜치 전략 및 커밋 컨벤션

### 5-1. 브랜치 전략 (GitHub Flow 기반)

- `main`: 배포 가능한 안정적인 코드가 유지되는 브랜치
- `develop`: (필요시) 다음 배포를 위해 기능이 통합되는 브랜치
- `feature/{기능명}`: 새로운 기능을 개발하는 브랜치 (예: `feature/ocr-integration`)
- `bugfix/{버그명}`: 버그 수정 브랜치
- `hotfix/{이슈명}`: 상용 배포 후 긴급 수정 브랜치

### 5-2. 커밋 메시지 컨벤션

Conventional Commits 형식을 따릅니다.

```
타입(스코프): 제목

본문 (선택)

꼬리말 (선택)
```

**타입 목록:**

- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정 (README 등)
- `style`: 코드 포매팅, 세미콜론 누락 등 (코드 논리 변경 없음)
- `refactor`: 코드 리팩토링
- `test`: 테스트 코드 추가 및 수정
- `chore`: 빌드 업무 수정, 패키지 매니저 설정 등

**예시:**

> `feat(api): 영수증 OCR 스캔 및 정규화 기능 추가`
> `fix(web): 1인분 레시피 분량 조절 버그 수정`

### 5-3. 커밋 원칙

- **단위 커밋(Single Responsibility)**: 반드시 **기능 하나를 개발할 때마다 개별적으로 커밋**해야 합니다.
- 여러 기능이나 서로 무관한 변경 사항을 하나의 커밋에 모아서 올리지 마세요. (예: UI 변경과 API 수정을 한 커밋에 담는 것 금지)
- 의미 있는 작은 단위로 커밋하여, 코드 리뷰를 용이하게 하고 문제 발생 시 롤백과 추적이 쉽도록 유지합니다.

## 6. 테스트 및 배포

### 6-1. 테스트

- 백엔드는 Jest를 활용하여 핵심 비즈니스 로직(Service)에 대한 단위 테스트를 필수적으로 작성합니다.
- 프론트엔드는 React Testing Library를 활용해 주요 사용자 흐름에 대한 테스트를 권장합니다.

### 6-2. CI/CD 파이프라인

- GitHub Actions를 활용하여 PR 생성 시 자동으로 린트 및 테스트 코드가 실행되도록 구성합니다.
- Turborepo의 캐싱 기능을 이용해 빌드 및 테스트 시간을 최소화합니다.
- 배포는 AWS EC2, S3, CloudFront를 활용하며 `main` 브랜치 머지 시 자동 배포가 되도록 구축합니다.
