# 🍱 defree — 서비스 기획안

---

## 1. 프로젝트 개요

> **영수증 한 장으로 시작하는 냉장고 관리 — 식재료 낭비 없는 자취 생활을 만들어드립니다.**

### 프로젝트 이름

**defree** — *"degree(정도·적정량)"* 와 *"free(자유)"* 를 합친 이름으로, **필요한 만큼만, 낭비 없이 사용한다**는 서비스 철학을 담았습니다. 냉장고 속 썩어가는 식재료를 바라볼 때의 죄책감, 오늘 뭐 먹을지 고민하는 피로감으로부터 사용자를 해방시키고, 적정량을 지키는 것이 곧 자유로운 자취 생활의 시작임을 지향합니다.

### 목표

- 식재료 구매부터 소진까지 전 과정을 자동화하여 자취생의 음식물 쓰레기와 불필요한 식비 지출을 줄인다
- 유통기한 알림과 레시피 추천을 통해 매일 앱을 열게 만드는 생활 밀착형 서비스를 구현한다
- 커머스 딥링크 연동을 통한 지속 가능한 수익 모델(BM)을 확보한다

### 타겟 사용자

| 구분 | 내용 |
|------|------|
| 1차 타겟 | 자취 초보, 요리 경험이 적은 20대 사회 초년생 |
| 2차 타겟 | 식비 절감에 관심 있는 1인 가구 전반 |
| 공통 특성 | 디스코드·슬랙 사용에 익숙, 스마트폰 중심 생활, 요리보다 편의성을 우선시 |

## 2. 문제 정의

### 1인 가구가 겪는 핵심 문제

**① 식재료 낭비와 죄책감**
된장찌개 한 끼를 위해 구입한 애호박·양파가 냉장고에서 썩어가는 상황이 반복된다. 소량 구매가 어려운 구조 탓에 1인 가구는 항상 과잉 구매와 잔여 식재료 처리 사이에서 소비를 낭비한다.

**② "오늘 뭐 먹지?" 결정 장애**
냉장고 안에 뭐가 있는지 모르는 채 배달앱을 켜는 악순환이 반복된다. 재고 파악 → 레시피 탐색 → 조리 결정까지의 마찰이 너무 크다.

**③ 장보기의 비효율**
지난주에 남은 고추장은 생각하지 못한 채 또 고추장을 집어 드는 중복 구매, 할인 타이밍을 놓친 제값 구매가 반복된다.

### 이 서비스가 해결하는 것

| 문제 | 해결 방식 |
|------|-----------|
| 식재료 낭비 | 유통기한 D-2 알림 + 잔여 재료 기반 레시피 즉시 제안 |
| 결정 장애 | 보유 재료로 만들 수 있는 요리를 자동으로 필터링하여 추천 |
| 중복 구매 | 현재 재고 분석 기반 스마트 장바구니 자동 생성 |
| 재고 입력 귀찮음 | 영수증 OCR 스캔으로 구매 즉시 자동 등록 |

## 2-1. 1인 가구 설계 전제

> defree는 처음부터 1인 가구만을 위해 설계된다. 일반 레시피 서비스의 기본값(2~4인분, 대용량 구매, 주 1회 장보기)은 1인 가구에게 맞지 않는다. 아래 전제들이 기능 설계 전반에 반영된다.

### 실제 1인 가구의 맥락

혼자 사는 사람의 저녁 결정은 "오늘 뭐 해먹지?"가 아니다.  
실제로는 **"오늘도 배달 시킬까 말까"** 다.

요리의 동기 자체가 낮고, 냉장고에 재료가 있어도 귀찮으면 배달을 시킨다. defree는 이 맥락을 정면으로 공략한다. **"배달비 아끼고 냉장고 털기"** 라는 프레임이 추상적인 건강·절약 메시지보다 훨씬 직접적으로 작동한다.

> 알림 톤 예시 비교
> - ❌ 일반 서비스: *"유통기한이 임박한 재료가 있어요. 레시피를 확인하세요."*
> - ✅ defree: *"방울토마토 오늘 안 쓰면 버려요. 배달비 3,000원 아끼고 10분 파스타 어때요?"*

---

### 4가지 1인 가구 설계 원칙

#### 원칙 1. 레시피 기본값은 항상 1인분

일반 레시피 서비스는 2~4인분을 기본으로 제공한다. 1인 가구가 이를 그대로 따르면 재료가 남고, 남은 재료는 다시 낭비로 이어진다.

- 레시피 추천 시 **1인분 분량을 기본값**으로 고정
- 레시피 카드에 인분 수 조절 UI 제공 (1인 → 2인 변경 가능하나, 기본은 1인)
- 만개의 레시피 API 연동 시 1인분 필터 우선 적용, 없을 경우 GPT로 분량 자동 환산

```
[레시피 카드 표시 예시]
방울토마토 파스타   ⏱ 10분   👤 1인분 (기본)
재료: 방울토마토 8개 · 파스타면 80g · 올리브유 1T
```

---

#### 원칙 2. 장바구니 추천은 소포장·소용량 기준

1인 가구의 최대 적은 대용량 묶음 상품이다. 양파 1.5kg을 혼자 소진하려면 2주가 걸리고, 그 사이 절반은 버려진다.

- 장바구니 추천 품목에 **소포장·낱개 상품 우선 표시**
- 커머스 딥링크 연동 시 소용량 필터 자동 적용 (예: 양파 낱개, 삼겹살 200g 소포장)
- 추천 카드에 **"1인 적정량"** 배지 표시

```
[장바구니 추천 예시]
✅ 양파 낱개 (1인 적정량)  →  쿠팡 최저가 보기
❌ 양파 1.5kg (대용량)      →  표시 안 함 (기본)
```

---

#### 원칙 3. 유통기한 알림 주기는 1인 소비 속도 기준

2인 가구는 두부 한 모를 3일 안에 소진하지만, 1인 가구는 5~7일이 걸린다. 같은 D-2 알림이라도 1인 가구에게는 이미 늦을 수 있다.

- 1인 소비 속도 기준 카테고리별 **알림 선행일 자동 조정**

| 카테고리 | 2인 기준 알림 | 1인 기준 알림 (defree) |
|----------|-------------|----------------------|
| 잎채소 (시금치·상추) | D-1 | **D-2** |
| 두부·계란 | D-2 | **D-3** |
| 육류 (돼지·닭) | D-1 | **D-2** |
| 근채류 (감자·양파) | D-3 | **D-4** |

- 사용자 실제 소비 패턴 학습 후 알림 시점 자동 보정 (향후 개선)

---

#### 원칙 4. 1인 소비량 기반 경고 메시지

재료의 유통기한뿐 아니라, **혼자서 소진 가능한지** 를 판단해 경고한다.

- 재고 등록 시 품목·수량·유통기한을 기반으로 1인 소비 가능 여부 계산
- 소진이 어렵다고 판단되면 경고 카드 노출

```
[1인 소비량 경고 예시]
⚠️  두부 2모 · 남은 기한 4일
    혼자 다 드시려면 4일 안에 요리 4번 필요해요.
    지금 냉동 보관하거나 오늘 두부 요리 2개 해두는 건 어때요?
    → [두부 요리 몰아보기]
```

- 계산 로직: `(재고 수량 × 1인분 소모량) ÷ 유통기한 잔여일 > 1회/일` 이면 경고 발동
- 냉동 보관 가능 품목은 냉동 안내 + 냉동 후 유통기한 재설정 유도

## 3. 주요 기능

### 3-1. 스마트 영수증 스캔 & 재고 자동화

> 사용자가 직접 입력하는 번거로움을 최소화하여 서비스 진입 장벽을 낮추는 핵심 기능

- **OCR & NLP 파이프라인**: Google Vision API / Clova OCR로 영수증 텍스트 추출 → LLM(GPT)을 거쳐 난해한 상품명(예: `삼양)라120*5`)을 표준 품목명(`라면`)으로 정규화
- **재고 자동 생성**: 정규화된 품목에 카테고리별 표준 유통기한을 매칭하여 구매일 기준 권장 소비 기한 자동 설정
- **디지털 영수증 연동 (확장)**: 쿠팡·마켓컬리·B마트 주문 내역 페이지 연동으로 온라인 구매 이력 자동 수집

---

### 3-2. '잔반 제로' 지능형 레시피 추천

> 냉장고 재료를 빠짐없이 소진하게 만드는 소비 촉진 기능

- **필터링 엔진**: 보유 재료 & 소스(양념) 데이터를 대조하여 추가 구매 없이 바로 가능한 요리 우선 노출
- **1인분 기본값 고정**: 모든 레시피 추천은 1인분 분량 기준, 만개의 레시피 API에 1인분 필터 우선 적용하며 없을 경우 GPT로 자동 환산
- **연쇄 소비 로직 (Chain-Recommendation)**: 오늘 저녁 메뉴 + 남은 자투리 재료를 활용한 내일 아침 메뉴를 세트로 연속 제안
- **외부 콘텐츠 연동**: YouTube Data API / 만개의 레시피 API 연동으로 검증된 조리법 제공
- **"배달비 아끼기" 프레임**: 레시피 카드에 예상 절감 배달비 표시 (예: *"배달비 3,500원 아끼기"*) — 요리 동기가 낮은 1인 가구의 행동을 유발하는 직접적 메시지

---

### 3-3. 멀티 채널 유통기한 리마인더

> 앱을 켜지 않아도 일상 속에서 식재료 상태를 인지하게 만드는 리텐션 강화 기능

- **1인 소비 속도 기준 알림 선행일 조정**: 카테고리별로 2인 가구보다 알림을 1~2일 앞당겨 발송 (예: 두부 D-3, 잎채소 D-2) — 혼자 먹으면 소진이 느리기 때문
- **1인 소비량 초과 경고**: 수량 대비 유통기한이 촉박한 경우 *"혼자 다 드시려면 X일 안에 요리 N번 필요해요"* 메시지 함께 발송
- **재고 차감 로직**: 알림 → 레시피 확인 → `조리 완료` 클릭 시 사용된 재료 재고에서 자동 차감
- **멀티 채널 지원**: Discord / Slack Webhook + FCM 모바일 푸시로 사용자 선호 채널에 알림 전송
  > 알림 예시: *"방울토마토 오늘 안 쓰면 버려요. 배달비 3,000원 아끼고 10분 파스타 어때요? 👉 [레시피 링크]"*

---

### 3-4. 스마트 장바구니 가이드

> 재구매 시점에 불필요한 중복 구매를 막고 지출을 최적화하는 의사결정 지원 기능

- **재고 분석 알고리즘**: 현재 남은 재료 + 소비 패턴 분석 → 이번 주 꼭 필요한 품목 리스트 자동 생성
  > 추천 예시: *"남은 고추장·참기름을 활용하려면 감자와 양파만 사세요!"*
- **소포장·소용량 우선 추천**: 1인 가구 기준 적정 용량 상품 우선 표시, 대용량 묶음 상품은 기본적으로 숨김 처리 (예: 양파 낱개 추천, 1.5kg 제외)
- **"1인 적정량" 배지**: 추천 품목 카드에 1인 가구 기준 적정 구매량 명시 (예: *"혼자라면 이 정도면 충분해요"*)
- **최저가 추적**: 플랫폼별(쿠팡, 마켓컬리 등) 할인 정보 수집 → 소포장 기준 최저가 제안
- **커머스 딥링크**: 추천 품목 클릭 시 소용량 필터 적용된 최저가 판매 페이지로 바로 연결 (제휴 수익 모델)

### 모노레포 구성 (Turborepo + pnpm)

> 백엔드(NestJS) · 프론트엔드(React) · 공통 모듈을 단일 레포에서 관리하여 타입 공유, CI/CD 일원화, 의존성 관리를 단순화한다.

#### 폴더 구조

```
defree/                          ← 루트 (Turborepo)
├── apps/
│   ├── api/                     ← NestJS 백엔드 (REST API + 스케줄러)
│   └── web/                     ← React PWA (프론트엔드)
├── packages/
│   ├── shared-types/            ← 공통 DTO / 인터페이스 (IngredientDto, RecipeDto 등)
│   ├── ocr/                     ← OCR + LLM 정규화 파이프라인 (독립 모듈)
│   └── notification/            ← Discord / Slack / FCM 웹훅 모듈
├── docker-compose.yml           ← PostgreSQL + Redis 컨테이너 통합 관리
├── turbo.json                   ← Turborepo 빌드 파이프라인 설정
└── pnpm-workspace.yaml          ← pnpm 워크스페이스 선언
```

#### 패키지별 역할

| 패키지 | 설명 | 비고 |
|--------|------|------|
| `apps/api` | NestJS 백엔드, 재고·레시피·알림 스케줄러 | `packages/*` 모두 의존 |
| `apps/web` | React PWA, Service Worker 기반 Web Push | `shared-types` 의존 |
| `packages/shared-types` | 백엔드↔프론트 공통 DTO, 인터페이스, 열거형 | 타입 불일치 방지 |
| `packages/ocr` | Clova OCR 호출 → GPT 품목명 정규화 파이프라인 | 추후 독립 서비스로 분리 가능 |
| `packages/notification` | Discord·Slack Webhook + FCM Web Push 발송 공통 모듈 | 알림 채널 확장 시 이 패키지만 수정 |

#### 도구 선택 이유

| 도구 | 이유 |
|------|------|
| **Turborepo** | 빌드 캐싱으로 CI 속도 향상, NestJS + React 조합에 설정 간단 |
| **pnpm workspaces** | 패키지 간 의존성 효율적 관리, Turborepo와 궁합 좋음 |
| **Docker Compose (루트)** | api · postgresql · redis 를 단일 `docker-compose.yml`로 통합 관리 |

## 5. 구현 방식

### 기술 스택 제안

| 영역 | 기술 | 선택 이유 |
|------|------|-----------|
| 백엔드 API | NestJS (Node.js / TypeScript) | 빠른 개발 속도, 데코레이터 기반 구조로 Spring과 유사한 아키텍처, @nestjs/schedule로 스케줄러 내장 지원 |
| 데이터베이스 | PostgreSQL | 재고·레시피 관계형 데이터 관리에 적합, 무료 오픈소스로 비용 절감 |
| 캐시 / 스케줄링 | Redis | 알림 큐 관리 및 레시피 결과 캐싱으로 API 호출 비용 절감 |
| OCR | Naver Clova OCR | 한글 영수증 인식 정확도 우수, Google Vision 대비 한국어 특화 |
| 텍스트 정규화 | OpenAI GPT (gpt-4o-mini) | 비정형 상품명의 표준화 처리, 저렴한 mini 모델로 비용 최소화 |
| 레시피 콘텐츠 | 만개의 레시피 API + YouTube Data API | 한식 레시피 DB 최대 규모, 영상 조리법 연동으로 콘텐츠 풍부화 |
| 알림 | Discord Webhook / Slack Webhook / FCM Web Push | 별도 인프라 없이 무료 멀티 채널 알림, FCM으로 웹앱에서도 브라우저 푸시 알림 지원 |
| 프론트엔드 | React (PWA) | 웹앱 단일 코드베이스로 빠른 배포, Service Worker + Web Push API로 앱과 유사한 알림 경험 제공 |
| 배포 | AWS EC2 | 안정적인 운영 환경, 추후 스케일업 용이 |
| 데이터베이스 | PostgreSQL + Redis (Docker 컨테이너) | EC2 위에서 Docker로 직접 운영, RDS·ElastiCache 비용 제거 |
| 도메인 (MVP) | DuckDNS | 무료 도메인으로 MVP 단계 비용 절감, 추후 유료 도메인으로 전환 가능 |

### 주요 구현 포인트

#### 1. OCR & 정규화 파이프라인
```
영수증 이미지 업로드
    → Clova OCR: 텍스트 추출
    → GPT (gpt-4o-mini): 비정형 상품명 → 표준 품목명 변환
        예) "삼양)라120*5" → "라면 (삼양, 5개입)"
    → 카테고리 매핑 테이블: 품목 → 카테고리 → 표준 유통기한 조회
    → PostgreSQL: 재고 INSERT (품목명, 수량, 구매일, 소비기한)
```

#### 2. 유통기한 추정 전략
- 카테고리별 권장 소비 기한 테이블을 DB에 사전 정의
  (예: 잎채소 3일, 근채류 7일, 육류 3일, 가공식품 제품 표기일 기준)
- 정확한 유통기한이 없는 경우 카테고리 기본값 적용, 사용자 수동 수정 가능

#### 3. 레시피 추천 엔진 (Chain-Recommendation)
```
현재 재고 목록 조회
    → 소진 우선순위 정렬 (유통기한 임박 순)
    → 만개의 레시피 API: 재료 태그 기반 레시피 검색
    → 오늘 저녁 메뉴 선정
    → 사용 후 남는 자투리 재료 추출
    → 내일 아침 활용 레시피 연쇄 추천
```

#### 4. 알림 스케줄러 & Web Push
```
@nestjs/schedule: 매일 오전 9시 전체 재고 D-day 계산
    → D-2 이하 재료 목록 추출
    → 해당 재료 기반 레시피 링크 생성
    → 사용자 설정 채널로 Webhook / FCM Web Push 발송

[Web Push 흐름]
Service Worker 등록 (브라우저 최초 접속 시)
    → FCM Web Push 구독 토큰 발급 → 서버 저장
    → 스케줄러 발동 시 FCM → 브라우저 Service Worker → 알림 표시
    ※ Android Chrome / 데스크탑 브라우저 완전 지원
    ※ iOS는 Safari 16.4 이상 + 홈 화면 추가(PWA) 필요
```

#### 5. 스마트 장바구니 알고리즘
```
현재 재고 현황 + 최근 30일 소비 패턴 분석
    → 소진 예정 품목 예측
    → 이번 주 구매 필요 품목 리스트 생성
```

### 향후 개선 방향

- **최저가 추적 & 커머스 딥링크**: 플랫폼별(쿠팡, 마켓컬리 등) 할인 정보 수집 → 장바구니 추천 품목에 최저가 구매 링크 연결, 제휴 수익 모델(BM) 확보
- **마트 앱 연동 고도화**: 쿠팡·마켓컬리·B마트 디지털 영수증 자동 수집 파이프라인 확장 (웹훅 또는 스크래핑)
- **AI 개인화**: 사용자별 선호 식재료·기피 음식 학습 → 맞춤 레시피 추천 정확도 향상
- **식단 플래닝**: 주간 영양 균형을 고려한 자동 식단표 + 가계부 연동 식비 리포트 제공
- **편의점 특화 카테고리**: 편의점 1+1 상품 활용법, 유통기한 임박 상품 당일 레시피 큐레이션

| 항목 | 서비스 | 비용 | 비고 |

| 서버 (백엔드) | AWS EC2 t2.micro | **무료** | 프리티어 12개월 무료, 750시간/월 |
| 데이터베이스 | PostgreSQL (Docker 컨테이너, EC2 내 운영) | **무료** | RDS 대신 EC2에서 직접 운영하여 비용 제거 |
| Redis | Redis (Docker 컨테이너, EC2 내 운영) | **무료** | ElastiCache 대신 EC2에서 직접 운영하여 비용 제거 |
| OCR | Naver Clova OCR | **무료** | 월 1,000건 무료 제공 |
| LLM 정규화 | OpenAI gpt-4o-mini | **~$1~3/월** | 영수증 1건당 약 200~500 tokens, 1,000건 기준 |
| 레시피 API | 만개의 레시피 API | **무료** | 파트너 협의 또는 공개 API |
| 동영상 연동 | YouTube Data API | **무료** | 일 10,000 quota 무료 제공 |
| 알림 (Discord/Slack) | Webhook | **무료** | 별도 인프라 불필요 |
| 알림 (Web Push) | FCM (Firebase) | **무료** | 건수 제한 없음, 웹앱 브라우저 푸시 지원 |
| 프론트엔드 배포 | AWS S3 + CloudFront | **무료** | 프리티어 S3 5GB, CloudFront 1TB 무료 |
| 도메인 (MVP) | DuckDNS | **무료** | 무료 서브도메인 제공, HTTPS는 Let's Encrypt 적용 |

> 프리티어 만료 후 예상 월 비용: EC2 ~$8 = **약 $8~11/월** (DB는 계속 Docker로 운영)

### 비용 최적화 전략

- **AWS 프리티어 최대 활용**: EC2 + S3 + CloudFront 프리티어 조합으로 12개월간 서버 비용 거의 무료
- **PostgreSQL · Redis Docker 컨테이너 운영**: RDS · ElastiCache 대신 EC2 위에서 Docker Compose로 직접 운영하여 DB 비용 완전 제거
- **Redis 캐싱으로 LLM 호출 최소화**: 동일 품목명 정규화 결과를 캐싱하여 반복 API 호출 제거
- **배치 처리로 토큰 비용 절감**: 영수증 품목 정규화를 건별이 아닌 묶음 처리(batch prompt)로 처리
- **DuckDNS + Let's Encrypt**: MVP 단계에서 도메인·SSL 비용 없이 HTTPS 운영 가능
- **스케일업 시점**: 프리티어 만료 또는 MAU 1,000명 초과 시 EC2 스펙 재검토, 트래픽 급증 시 DB를 RDS로 이전 고려
